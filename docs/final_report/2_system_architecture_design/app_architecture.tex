\section{Высокоуровневая архитектура приложения}

Система TechEventsHub спроектирована по принципу сервис-ориенти-рованной архитектуры: функциональность разделена между тремя самостоятельными компонентами, каждый из которых обладает чётко ограниченной зоной ответственности и взаимодействует с остальными исключительно через API. 
Такое разграничение позволяет масштабировать компоненты независимо друг от друга, модифицировать их реализацию без затрагивания смежных модулей и упрощает тестирование каждой части системы в изоляции от остальных.

\subsection*{Подсистема сбора и агрегации данных (Core API)}

Центральным компонентом системы является серверное приложение, реализованное на основе фреймворка FastAPI. 
Оно является единой точкой входа для всех клиентских запросов и включает в себя несколько функциональных подсистем. 

Слой управления подписками отвечает за регистрацию отслеживаемых ресурсов и хранение пользовательских фильтров в базе данных PostgreSQL. 

Подсистема идентификации обеспечивает авторизацию пользователей через протокол Yandex OAuth с выдачей маркеров доступа JWT. 
Также эта подсистема поддерживает связывание учётных записей двух типов -- веб-аккаунта и учётной записи Telegram в рамках единой пользовательской сущности. 

Планировщик задач осуществляет периодический обход зарегистрированных для отслеживания ресурсов: для Telegram-каналов применяется протокол MTProto, для веб-страниц -- синтаксический анализ HTML-разметки. 

Извлечённый текст передаётся во внешний сервис YandexGPT, который формирует структурированное описание события с указанием даты, места проведения, формата и цены. 

Полученные данные сопоставляются с индивидуальными фильтрами каждого пользователя, и уведомление направляется только тем пользователям, условия которых были соблюдены. 

Маршрутизатор уведомлений определяет канал доставки для каждого пользователя в соответствии с его настройками -- в мессенджер Telegram, на адрес электронной почты или в оба канала одновременно.

\subsection*{Клиентское веб-приложение}
Для обеспечения удобного доступа к системе через браузер в рамках данной работы был разработан пользовательский интерфейс на основе фреймворка Next.js с использованием серверного рендеринга. 

Приложение реализует полный цикл взаимодействия с пользователем: авторизацию через Yandex OAuth, управление подписками на источники анонсов, выбор фильтров по различным критериям и настройку параметров уведомлений. 

Маркер доступа (JWT), получаемый по завершении процедуры авторизации, сохраняется в защищённой от клиентских сценариев cookie-записи.
Это исключает его несанкционированное считывание из браузерного окружения. 
Сохранение пользовательских настроек и обновление данных происходит не напрямую из браузера, а через сервер веб-приложения. 
Он выступает в роли посредника, который самостоятельно связывается с Core API.

Безопасность закрытых разделов обеспечивается предварительной проверкой прав доступа пользователя. 
Система проверяет наличие авторизации до того, как начнет формировать страницу. 
Если пользователь не авторизован, запрос немедленно отклоняется, а интерфейс возвращает его на главную страницу.

\subsection*{Программный агент (Telegram-бот)} 
Telegram-бот предоставляет альтернативный текстовый интерфейс для пользователей, предпочитающих читать анонсы в мессенджере. 
Агент обрабатывает команды управления подписками в форме диалога: после ввода адреса ресурса пользователю предлагается последовательно уточнить критерии фильтрации.

Помимо функций ввода команд, бот принимает входящие уведомления об обнаруженных событиях, поступающие от подсистемы сбора данных, и доставляет их конечным пользователям. 
Связь программного агента с Core API осуществляется по протоколу HTTP с передачей идентификатора чата в заголовке запроса, что позволяет серверному приложению однозначно определить пользователя без использования маркера доступа.

\subsection*{Межсервисное взаимодействие} 
Система сочетает два режима обмена данными. 
Синхронное взаимодействие реализовано посредством REST-запросов по протоколу HTTP: оба клиентских компонента обращаются к Core API для выполнения операций с подписками и профилем пользователя и получают ответ в рамках того же сеанса соединения. 

Асинхронная доставка уведомлений реализована через брокер сообщений Apache Kafka. Подсистема сбора данных публикует сообщение о новом событии в очередь, а программный агент подписан на соответствующий топик и обрабатывает сообщения по мере их поступления. 
Такой подход позволяет модулям работать независимо друг от друга. 
Например, если программный агент перезагружается или временно недоступен, подсистема сбора данных не простаивает в ожидании ответа. 
Все уведомления сохраняются в очереди их доставка пользователям возобновляется сразу после восстановления связи.
В случае отказа брокера сообщений подсистема автоматически переходит на прямую передачу уведомления программному агенту по HTTP.
